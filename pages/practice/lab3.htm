Лабораторная работа 3: «Создание интерактивного электронного технического руководства»


TG Builder - это система автоматизированной подготовки интерактивных электронных технических руководств. Функциональность TG Builder обеспечивает возможности работы коллектива авторов, планирования работ и обмена данными между разработчиками, кроме того, использование СУБД Oracle для хранения подготовленной документации обеспечивает возможность подготовки руководств огромного размера.

TG Builder - это также тесная интеграция с PDM системами и различными офисными приложениями, установленными на Вашем предприятии.

Возможности синхронизации между PDM и TG Builder позволяют в значительной степени уменьшить объем работ и вероятность внесения в руководство некорректных данных.

Основные характеристики TG Builder:

• Подготовленные руководства соответствуют стандартам ISO 8879 SGML, MIL 87269, ГОСТ Р 50.1.

• Возможность работы над руководством коллектива авторов, включая планирование работ и обмен информацией по сети.

• Четкое разделение прав пользователей, контроль доступа к данным.

• Централизованное администрирование системы.

• Интеграция с PDM системами и офисными приложениями.

• Возможность использования в руководствах помимо текстовых и графических данных аудио- и видео- роликов и трехмерных моделей.


Цель работы:


На основе бумажного руководства по эксплуатации в системе автоматизированной подготовки интерактивных электронных технических руководств TG Builder создает электронную версию руководства по эксплуатации.

На основе оглавления бумажного руководства разрабатывается структура электронного руководства.

После того как разработана структура, приступаем к редактированию глав руководства.

1. Работа с текстом. Информационное наполнение разделов электронного руководства может быть осуществлено следующими способами: или непосредственная перепечатка текста из бумажного руководства в текстовом редакторе TG Builder 4.0.2.112, или сканирование текста бумажного руководства с последующим распознаванием текста и вставкой этих отсканированных фрагментов в редактор TG Builder, где возможно их редактирование; или вставка текста из готовых файлов.

2. Работа с изображениями. Как и при работе с текстом изображения могут быть получены путем сканирования и вставлены в электронное руководство. Одной из задач при работе с текстом является создание ссылок с фрагментов текста на другие разделы руководства, а при работе с изображениями простановка ссылок с частей изображения на другие разделы руководства и организация так называемых карт ссылок. В процессе работы над ру- ководством можно осуществлять предварительный просмотр созданных частей руководства. После того, как руководство создано, в TG Builder предусмотрена возможность импортировать некоторые разделы или все руководство в один из двух форматов - это формат HTML (просмотр осуществляется стандартными средствами - MS Internet Explorer или Netscape Communicator) или бинарный формат (сопровождается индивидуальным средством просмотра).

2. Присваивание типа модулю данных.


Модули данных подразделяются на типы в зависимости от вида технических сведений, которые в них содержатся:

• Описательный

• Процедурный

• Электронный каталог

• Ресурсы и сроки службы

• Формы технического обслуживания

• Перечень работ по техническому обслуживанию

• Процедура поиска и устранения неисправностей

Для задания типа и наименования Модуля данных:

1. Выделим первый Модуль данных в структуре Шасси.

2. Нажмем на кнопку Тип модуля данных в правом нижнем углу диалогового окна Структура раздела.

3. Выберем тип модуля данных Описательный.

4. В поле Раздел введем наименование модуля данных – Шасси – Описание. По стандарту ASD S1000D наименование МД должно состоять из двух частей, разделяемых дефисом, каждая из которых начинается с заглавной буквы.

5. Нажмем клавишу Enter.

Подобным образом присвоим тип и наименование другому МД, задав Процедурный МД.

3. Кодирование элементов проекта.


По стандарту ASD S1000D МД присваивается стандартизированный идентификатор, который называется кодом МД.

Рассмотрим процесс присваивания кодов элементам структуры проекта на примере системы Шасси демонстрационного проекта. В правой нижней части окна Структура раздела находятся инструменты, позволяющие вводить и запоминать коды. При использовании инструмента Установить код происходит запоминание кода для данного элемента структуры, а при использовании инструмента Очистить код удаляются коды из полей.

Значения кодов вводятся в поля, которые находятся в нижней левой части диалогового окна Структура раздела.


   

Рис. 3.18. Поля кода МД.   


Для элемента Система предоставляется 4 поля для ввода значений кода. Первое поле – Код конфигурации изделия(авто). Три следующих поля предназначены для ввода значений кода стандартной системы нумерации(SNS).Для выбора значения кода системы из списка:

• Нажмем правой кнопкой мыши на область второго поля и выберем из списка название системы Шасси. При этом в поле появится значение кода 32.

• В третье и четвертое поля введем по два нуля, так как если используется вся система, то значение кода для подсистемы равно 0, для под-подсистемы – 0 и для агрегата-00.

• После ввода значений во все поля нажмем Enter и код установиться.


   

Рис. 3.19. Код стандарта ASD S1000D МД.   


Аналогичным образом установим код для элемента Гидравлическая система. Полученный код будет: А-29-00-00. В структуре проекта выберем подсистему Основная опора шасси, входящую в систему Шасси.

В коде подсистему уже имеется значение кода системы Шасси, к которой относится данная подсистема. Для ввода кода подсистемы доступно одно поле. Это второе поле стандартной системы конфигурации(SNS). Нажмем по нему правой кнопкой мыши и выберем Основная опора шасси. При этом появится значение кода 10.


   

Рис. 3.20. Элементы поля стандартной системы конфигурации(SNS).   


Аналогичным образом заполняем остальные коды подсистем. Коды подсистем будут следующими: А-32-20-00; А-32-30-00; А-32-40-00; А-32-50-00; А-32-60-00.

Выделим под-подсистему Стойка основной опоры шасси, входящую в подсистему Основная опора шасси.

Код для под-подсистем назначается разработчиком из интервалов между значениями кодов подсистем. В данном примере подсистема Основная опора шасси с кодом 10 будет содержать две под-подсистемы со значениями кода 1 и 2.


   

Рис. 3.21. Код стандарта ASD S1000D МД.   


Выделим деталь Амортизатор основной опоры шасси. Для ввода кода доступны два поля:

1. Код демонтажа - используется для кодирования компонента внутри изделия.(Введем значение - 01)

2. Вариант кода демонтажа – указывает на альтернативные составные части изделия.(Введем значение- А)

3. Нажмем Enter.

Аналогично для другого.


   

Рис. 3.22. Код демонтажа.   


Выделим МД Амортизатор основной опоры шасси – Заправка азотом. Для ввода кода предоставляется 12 полей часть из них уже заполнена.

1. В первое поле автоматически введен Идентификационный код модели, указан при создании проекта– BU.

2. Второе поле – код конфигурации изделия.

3. Три следующих поля (3-5) были заполнены при кодировании системы – 32-11-01.

4. Шестое поле Код демонтажа было заполнено при кодировании Амортизатор основной опоры шасси – 01.

5. Седьмое поле также закодировано ранее- А.

6. Поля 8-10 – Информационный код, который определяет тип информации в МД. Нажмем правой кнопкой мыши на поле 8 и выберем из списка название Обслуживание. Значение кода равно 2.

Нажмем правой кнопкой мыши и выберем Заправка. Значение кода равно 1.Нажмем правой кнопкой мыши на поле 10 и выберем Заправка азотом. Значение кода равно 4.

7. Поле номер 11 заполняется автоматически.

8. В поле 12 вводим А – для летательных аппаратов.
